# 07. テスト・リリース・ロールバック計画

## テスト方針

現行システムとの仕様互換を重視し、API、画面、DB、帳票、権限制御を段階的に検証します。

## テスト種別

| 種別 | 対象 | 主な確認 |
|---|---|---|
| 単体テスト | Application Service、Repository、帳票生成 | 入力チェック、業務判断、SQL、Excelセル設定 |
| APIテスト | ASP.NET Core Web API | ステータス、レスポンス、認証、認可、エラー形式 |
| 画面テスト | React | ルーティング、フォーム、一覧、ダウンロード |
| DB移行テスト | OracleからSQL Server | 件数、制約、文字化け、採番、集計一致 |
| 帳票比較テスト | 現行Excelと新Excel | セル値、書式、ファイル名、履歴 |
| セキュリティテスト | 認証認可、CSRF、CORS、ファイル取得 | 他人のデータ参照不可、トークン検証 |
| 性能テスト | 検索、帳票出力 | 大量データ、タイムアウト、メモリ使用量 |

## 重点テストケース

| 機能 | ケース |
|---|---|
| ログイン | 正常、パスワード誤り、未認証API、ログアウト後アクセス |
| ダッシュボード | 本日件数、今月時間、未出力件数、最近の活動 |
| 作業日報登録 | 必須、文字数、分類不正、0時間、24時間超、正常登録 |
| 作業実績検索 | 期間、社員名、部署名、分類、プロジェクト、一般ユーザー制限 |
| 月次帳票出力 | 管理者対象者指定、一般ユーザー自分のみ、0件出力、正常出力 |
| 帳票履歴 | 検索、詳細、成功時ダウンロード、ERROR/PROCESSING時ダウンロード不可 |
| ファイル安全性 | 保存領域外パス拒否、削除済みファイル、ファイル名サニタイズ |

## リリース前チェック

- SQL Serverバックアップを取得した。
- データ移行リハーサルを実施し、件数と主要集計が一致した。
- 採番シード値が既存最大IDより後ろになっている。
- 認証、認可、CSRF、CORS設定を検証した。
- 現行Excelと新Excelの比較を完了した。
- 帳票保存先の容量、権限、バックアップ方針を確認した。
- 環境変数、接続文字列、秘密情報がリポジトリ外で管理されている。
- 本番切替手順とロールバック手順を検証環境で通した。

## リリース手順案

1. 現行システムの利用停止時間を確定する。
2. Oracleのバックアップを取得する。
3. 帳票保存領域をバックアップする。
4. Oracleからデータを抽出する。
5. SQL Serverへデータを投入する。
6. 件数、制約、主要集計、サンプル帳票を検証する。
7. ASP.NET Core APIをデプロイする。
8. Reactをデプロイする。
9. 疎通、ログイン、主要業務、帳票出力を確認する。
10. 利用開始を案内する。

## 並行稼働案

| 案 | 内容 | 注意 |
|---|---|---|
| 一括切替 | 停止時間中にDB移行し、新システムへ切替 | 手順は単純だが戻し判断が重要 |
| 参照並行 | 現行を参照用に残し、新規登録は新システムのみ | 利用者導線の整理が必要 |
| 段階移行 | 機能単位で切替 | DB二重管理が複雑になりやすい |

初期案は一括切替です。機能数が比較的限定されており、DB二重書き込みの複雑さを避けられるためです。

## ロールバック方針

| 状況 | 方針 |
|---|---|
| 切替前検証で失敗 | 新システム公開を中止し、現行を継続 |
| 切替直後で新規データなし | DNS/URL/アプリを現行へ戻す |
| 切替後に新規データあり | SQL Server側の新規データをOracleへ戻せるか個別判断 |
| DBスキーマ不整合 | SQL Serverをバックアップ時点へ復元 |
| 帳票ファイル不整合 | 保存領域バックアップから復元 |

## ロールバック制約

- 新システムで登録された作業日報や帳票履歴をOracleへ戻す手順がない場合、完全ロールバックはできません。
- パスワードハッシュ方式や認証状態を変更した場合、ログイン周りの戻しに注意が必要です。
- 帳票ファイル名や保存先構造を変更した場合、履歴との整合性回復が必要です。

## 運用設計

- ログイン失敗、ログアウト、帳票出力、再ダウンロードを監査ログ候補にします。
- `PROCESSING` が一定時間を超えた履歴を検出する運用を作ります。
- 帳票保存先の保管期間、削除方針、バックアップ、復元手順を決めます。
- SQL Serverのバックアップ、インデックスメンテナンス、容量監視を設定します。
- 障害時の問い合わせに備え、`traceId` とアプリログを紐づけます。
